# アーキテクチャ設計判断記録

## ネットワーク設計

### なぜデフォルトVPCを使わなかったか
デフォルトVPCはCIDRやサブネット構成が固定されており、設計意図が見えない。
自分でVPCを作ることで「ネットワークを理解して設計した」ことを証明できる。

### なぜ /16 のCIDRを選んだか
- `/16` = 65,536 IPアドレス
- 実際に使うのは `/24`（256個）のサブネット1つだが、将来の拡張を考慮
- Phase2でプライベートサブネット（RDS用）を追加する際に余裕を持って切り出せる

### なぜプライベートサブネットを今回作らなかったか
- MVPのスコープを超えるため（学習コストより完成を優先）
- 設計上は「追加予定」として説明できる状態にしている

## セキュリティ設計

### SGでSSHを自分のIPのみに制限した理由
- SSH（ポート22）を `0.0.0.0/0` で開放すると、世界中からブルートフォース攻撃を受ける
- 実際にAWS上でSSHを全開放したサーバーには数分で攻撃が来る
- 最小権限の原則：必要な人・IPのみにアクセスを許可する

### IAM Roleを使ってアクセスキーをEC2に持たせなかった理由
- EC2にアクセスキーを直接配置すると、サーバー侵害時にキーも盗まれる
- IAM Roleはトークンを一時的に発行する仕組みのため、キー流出リスクがない
- AWSのベストプラクティスに準拠

## 運用設計

### CloudWatch Agentを入れた理由
- EC2のローカルログはインスタンス停止・削除で消える
- CloudWatch Logsに集約することで、EC2が消えてもログが残る
- 複数台になった場合でも一元管理できる

### S3のライフサイクルを90日に設定した理由
- 無期限保存するとコストが増加し続ける
- 障害調査で必要なログの参照期間は実務的に30〜90日が多い
- コストと実用性のバランスを取った判断
